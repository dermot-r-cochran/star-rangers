# Changelog

All notable changes to this project are documented in this file.

The format loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and version numbers follow [Semantic Versioning](https://semver.org/) — `MAJOR.MINOR.PATCH`, where `MINOR` covers backward-compatible additions (a new lore entry, a new deploy.conf key, a new theme) and `PATCH` covers fixes with no new surface area. Nothing in this project's history so far has required a `MAJOR` bump: every `deploy.conf` key, theme, and build output added has been additive, and every default has stayed backward compatible.

## [Unreleased]

### Added

- **Full-text search**, powered by [Pagefind](https://pagefind.app/): `npm run build` now runs `eleventy && pagefind --site _site`, indexing the built output. A hand-rolled header search box (`src/js/search.js`, styled in `main.css`'s new `/* --- Search --- */` section) uses Pagefind's JS API directly rather than its prebuilt `pagefind-ui` widget, so it matches the site's own dark theme instead of overriding another package's bundled CSS. `<main data-pagefind-body>` scopes indexing to actual page content; breadcrumbs, "back to X" footers, chapter pagination, the chapter-scenes list, and the comments widget all carry `data-pagefind-ignore` so search results aren't full of repeated nav boilerplate. The search widget's `data-pagefind-bundle` path carries the same `/star-rangers/` prefix as every other internal link, so `SITE_PATH_PREFIX` rewrites it automatically too. Verified end-to-end with a headless-browser test against a real built+indexed site: query → results with correct titles/excerpts/highlighting → click-through to the right page, at desktop and mobile viewport widths, with zero console/network errors.
- **`npm run new`** (`scripts/new-content.js`): scaffolds a new character, lore entry, codex entry, glossary entry, or chapter, prompting for each type's fields instead of requiring a contributor to memorize a layout's exact front-matter shape by copying an existing file. Chapter numbers auto-derive the `id`, directory, and filename, removing a whole class of copy-paste drift. Timeline entries aren't covered (see the script's own header comment for why).
- **Content validation** (`scripts/validate-content.js`, run by `npm test` before the Eleventy dry run): checks every content file's front matter against a shared schema (`lib/content-schema.js`, the same one `npm run new` scaffolds from) and fails with a clear per-file message — a missing required field, a non-numeric `season`/`episode`/`chapter`/`sort_order`, or a chapter whose `id` or filename disagrees with its own front matter — instead of a blank page or a cryptic Nunjucks error surfacing later. Verified against all 223 existing content files (passes clean) and against several deliberately-broken test files (each catches correctly), including through `npm test`'s actual short-circuit behavior on failure.
- **giscus-powered discussion comments** on every character, lore, glossary, codex, episode, and chapter/scene page, backed by GitHub Discussions on a separate, dedicated `Star-Rangers/sciencefiction-site-comments` repo (kept apart from this repo's own dev-facing Discussions). Config lives in `src/_data/giscus.js`; the embed is in `src/_includes/base.njk`, gated on new `comments`/`commentsCategory` front matter set by the six relevant layouts. Three page-mapped categories (**Characters**, **Lore & Worldbuilding**, **Episode Discussion**) get one auto-created discussion per page path; five more (**Announcements**, **General**, **Q&A**, **Theories & Predictions**, **Fan Creations**) are for open community discussion unrelated to any specific page. See the README's new "Discussion forum (giscus)" section for the full category list and the one-time GitHub/giscus.app setup steps needed before the repo/category IDs can be filled in.
- **`COMMENTS_ENABLED`** `deploy.conf`/`ALT_<id>_COMMENTS_ENABLED` key (default `true`) to build a clone with the giscus widget above turned off entirely — read as a plain `.eleventy.js` global-data toggle (`process.env.COMMENTS_ENABLED`), the same pattern `THEME` already uses. The GitHub Pages workflow sets it `false`, since Pages' `/star-rangers/`-prefixed URLs would otherwise create a second, disconnected set of pathname-mapped discussions alongside the root-served cPanel domains'. In `scripts/cpanel-deploy.sh`, `build_and_deploy()`'s existing 11 fixed positional settings weren't given a 12th: this and any future simple on/off setting is instead passed as a trailing `NAME=value` string, parsed by a small `case` inside the function (unrecognized names fail the deploy loudly) — keeps deploy.conf's growing option list from making that function's call sites harder to read or reorder correctly.
- **`scripts/fetch-giscus-ids.js`** (`npm run fetch-giscus-ids`): fetches the giscus repo ID and every configured category's ID from GitHub's GraphQL API in one call, using the repo/category names already declared in `src/_data/giscus.js`, and can patch them straight into that file with `--write` — an alternative to re-running the giscus.app wizard once per category.
- **"Fan fiction" section on the About page**, pointing readers who want to write their own Star Rangers stories to AO3 or Wattpad rather than this site or its comments repo, with a suggested tag so their work stays findable. The comments repo's **Fan Creations** category (see the giscus category table above) is for linking/discussing those works, not hosting them — no in-repo fic-hosting or forum was built for this, by design, per the tradeoffs discussed when the category list was drafted.

### Added

- **`FORKING.md`**: a step-by-step guide to forking this repo either as an original interactive fiction site (replace the content, keep the engine) or as a Star Rangers fan fiction clone (keep some/all content too, per the new Fan Works Policy). Covers rebranding, the `/star-rangers/` path prefix, standing up a separate comments repo, and deployment for both GitHub Pages and cPanel.
- **`SITE_PATH_PREFIX`** env var (`.eleventy.js`): rewrites every hardcoded `/star-rangers/` absolute link across layouts *and* content (~200 files) to a different base path at build time, via an Eleventy transform. Exists for forkers serving from a different subpath or domain root, without hand-editing every file it appears in; unset by default, which leaves this project's own output byte-for-byte unchanged. The GitHub Pages workflow now passes it through from an optional `vars.SITE_PATH_PREFIX` repository variable. cPanel deploys don't need it — `scripts/cpanel-deploy.sh` already strips the same prefix with its own post-build sed step, independently of this.
- **Fan Works Policy** in `CONTENT-LICENSE.md`: an explicit, standing exception to the story content's No-Derivatives term for non-commercial fan fiction, fan art, and fan fiction clones of this site (forking the engine and some or all of the existing content) — conditioned on staying non-commercial, clearly labeled as unofficial, and attributed back to the original. The About page's "Fan fiction" section and the site's own on-page licence section both now point to it.

### Changed

- **Split the repository's licence in two**, so the site engine can be forked without implying any rights over the story itself. `LICENSE` is now plain **MIT**, covering `.eleventy.js`, `lib/`, `src/_includes/`, `src/css/`, `src/js/`, and `scripts/`; the story content (`src/seasons/`, `src/threads/`, `src/characters/`, `src/timeline/`, `src/lore/`, `src/glossary/`, `src/codex/`, `story-bible/`, `prompts/`) keeps the original **CC BY-NC-ND 4.0** terms, now in a new `CONTENT-LICENSE.md`. Previously the repo-root `LICENSE` file applied CC BY-NC-ND's NoDerivatives term to the whole repository, code included, which technically forbade forking the engine at all — a mismatch with `src/index.md`'s own licence section, which already scoped its wording to "text and world-building content" only. `package.json`'s stale `"license": "ISC"` (never matched either license) is now `"MIT"` to match, and `src/index.md`'s licence section gains a short pointer to the code's separate MIT licence.
- **Storyline Threads** as a top-level site category (`/threads/`), grouping seasons under the independent narratives established in `story-bible/story-bible-summary.md`'s "Narrative Structure" section: **Founding Era** (Season 0) and **Tissadelle Shepherd's Arc** (Seasons 1, 3, 5). The canonical thread→season registry lives in the new `lib/storyline-threads.js` (re-exported as global template data via `src/_data/storylineThreads.js`, and as the new `threadForSeason` Nunjucks filter), so it's the single source of truth for both the site's grouping and `deploy.conf`'s new filter below. `/seasons/` now groups its season blocks under a thread heading instead of listing seasons flat, each season's own index page carries a "Part of: `<thread>`" link, and each thread gets its own page at `/threads/<id>/` listing the seasons it covers. A season not yet assigned to a thread falls back to an "Unsorted" grouping rather than silently disappearing.
- **`THREADS` `deploy.conf` key**, alongside the existing `CHARACTERS`/`TOPICS`: a comma-separated list of storyline thread ids that narrows a deployed clone to the seasons those threads cover (every chapter whose `season` falls under a listed thread, plus any lore/timeline/glossary/codex entry explicitly tagged with the thread id). Combines with `CHARACTERS`/`TOPICS` rather than replacing them — matching any one of the three includes a page. Wired through `lib/content-filter.js` (`getContentFilter`, `isChapterIncluded`'s new `isSeasonInIncludedThread` check), `scripts/cpanel-deploy.sh` (including its own `ALT_<id>_THREADS` per-alt-domain variant), and documented in `sample-deploy.conf`, `.cpanel.yml`, and the README.
- **"Narrative Structure — Multiple Independent Storylines"** section in `story-bible/story-bible-summary.md`: formalizes the series' two-level storyline pattern already implicit in the published content — a macro-level season spine (Tissadelle Shepherd's career across S1/S3/S5/S6-7, with Seasons 2 and 4 deliberately left open as slots for storylines that don't involve her) and a micro-level pattern of independent, non-overlapping POV threads within a season that converge at a planned point (Season 1's Threshold Station/Marsh Causeway threads; the Season 6-7 treatment's institutional/interior threads). Documents the five criteria that make two threads "independent" rather than just alternating scenes, and flags Seasons 3 and 5 as the current exception to the pattern rather than a second template.

Cross-linked update in `celtic-union-of-planets.md`. `kingdom-of-the-four-islands.md` gets a small clarifying addition (centuries-old, since-forgotten atmospheric engineering by the lost colony's founders) so it doesn't read as a second, unflagged exception to the same rule.
- `ADMIN_EMAIL` deploy.conf key now defaults to `admin@<DOMAIN>` instead of staying unset, so every cPanel clone gets deploy-log notifications out of the box without needing its own `deploy.conf` entry for it.

## [1.5.0] - 2026-07-09

### Added

- `scripts/generate-themes.js` (`npm run generate-themes`): generates every `src/css/theme-<name>.css` from `main.css` plus a small per-theme palette registry, so a theme can only ever differ from `main.css` in its `:root` custom properties, its five `.pov-block--<character>` colors, and `.character-badge--status-active` — everything else is guaranteed structurally identical to `main.css`.
- Four new standard, non-domain-specific themes: `light` (day mode), `high-contrast` (low-vision accessibility), `sepia` (e-reader tone), `solarized` (Solarized Dark).
- Theme catalog documented in `sample-deploy.conf` and `README.md`.

### Fixed

- The `pets` theme (undercover-pets.com) had drifted significantly out of sync with `main.css`: it was missing link-color styling across nearly every content body (`.chapter-body a`, `.codex-entry__body a`, `.character-body a`, `.glossary-entry__body a`, `.lore-entry__body a`, and `.character-portrait`), meaning most internal links on that production domain rendered in the browser's default blue instead of the theme's own accent color. The `fellowship` and `starquest` themes had smaller drift (missing the 1.2.0 link-color fallback and two newer component rules). All three are now regenerated in sync with `main.css` by `scripts/generate-themes.js`; their palettes are unchanged.

## [1.4.0] - 2026-07-09

### Added

## [1.2.0] - 2026-07-09

### Changed

- cPanel deploy (`scripts/cpanel-deploy.sh`) now fails the deployment — and triggers the existing `ADMIN_EMAIL` failure notification — if `.htaccess` or `.well-known/security.txt` are missing from the deployed destination, instead of only checking for `index.html`. cPanel is the only deploy target that reads `.htaccess` at all (GitHub Pages ignores it), so a silent regression there would previously have shipped with none of its security headers and still reported success.

### Notes

- Audited both deploy targets for security posture: cPanel's `.htaccess` already set CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy`; `.well-known/security.txt` was present and current. GitHub Pages has no supported mechanism for custom response headers at all, so it ships with none of these — a platform limitation, not a config gap.
