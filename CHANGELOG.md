# Changelog

All notable changes to this project are documented in this file.

The format loosely follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and version numbers follow [Semantic Versioning](https://semver.org/) — `MAJOR.MINOR.PATCH`, where `MINOR` covers backward-compatible additions (a new lore entry, a new deploy.conf key, a new theme) and `PATCH` covers fixes with no new surface area. Nothing in this project's history so far has required a `MAJOR` bump: every `deploy.conf` key, theme, and build output added has been additive, and every default has stayed backward compatible.

The second is the plain-voice telling — a contemplative answering a cadet at the Eden commissioning who asks why the Rangers sing an Old-Earth prophet's song ([Hymn of the Thin Places](src/codex/hymn-of-the-thin-places.md)), threading the twinned story to Krast without ever naming her. Both are filed in the Codex, not Lore, per the contested/interpretive rule (the mapping is one correspondent's, non-diegetic to the political record); the Cascade's no-moral-cosmology and "misattribution as divine will" cautions are carried through both, and the mythic overstatement that gives the MSC a founder it never had is itself flagged as an Arilon-adjacent effect. - **Multilingual approach: English as canon, browser translation, and human translations as fan works** — no official per-locale build, by design (an unverifiable translation on the canonical site would assert canon its author can't confirm, worst of all in the Codex/narrative where precise diction is the point). `src/_includes/base.njk`'s footer gains a labeled note (`.site-lang-note` in `main.css`, regenerated across every theme) telling readers their browser can translate any page but that such translations are automatic/**unofficial** and the English text is always authoritative on canon. `CONTENT-LICENSE.md`'s Fan Works Policy and the About page's Fan Works section (now anchored `#fan-works`) explicitly welcome human translations as non-commercial, clearly-labeled fan works. New Journal entry **`src/journal/a-canon-i-cannot-verify.md`, "A Canon I Can't Verify in Another Language"** records the reasoning.

The project's own comment repos are now a `GISCUS_PROFILES` registry in `giscus.js` (`default` → `sciencefiction-site-comments` for the full-site/reference domains incl. starquest.*; `church-space` → `churchspace-site-comments` for church-space.site/.online and their Fellowship of Light addon domains). A deploy selects one with a single `GISCUS_PROFILE` key instead of pasting six repo/category IDs — so `sample-deploy.conf`'s church-space example blocks collapse from ~7 giscus lines each to one, and a Fellowship of Light sample block was added alongside. An unknown profile name fails the build (typo guard) instead of silently using the default board. The six explicit `GISCUS_REPO`/`GISCUS_REPO_ID`/`GISCUS_CATEGORY_*_ID` keys remain as a per-field escape hatch (each overrides just its field of the selected profile). Separately, a **third-party fork** can now point comments at its own repo without editing engine code or `deploy.conf`: an untracked `giscus.local.json` (copy `sample-giscus.local.json`) is read at build time and becomes the fork's default board on every path — `npm start`, GitHub Pages, and any cPanel deploy that doesn't select a profile — with `GISCUS_PROFILE`/`GISCUS_*` env vars still taking precedence. Default (unset) behavior is unchanged: the same `sciencefiction-site-comments` board with the same IDs as before.

## [1.2.0] - 2026-07-09

### Changed

- cPanel deploy (`scripts/cpanel-deploy.sh`) now fails the deployment — and triggers the existing `ADMIN_EMAIL` failure notification — if `.htaccess` or `.well-known/security.txt` are missing from the deployed destination, instead of only checking for `index.html`. cPanel is the only deploy target that reads `.htaccess` at all (GitHub Pages ignores it), so a silent regression there would previously have shipped with none of its security headers and still reported success.

### Notes

- Audited both deploy targets for security posture: cPanel's `.htaccess` already set CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy`; `.well-known/security.txt` was present and current. GitHub Pages has no supported mechanism for custom response headers at all, so it ships with none of these — a platform limitation, not a config gap.
